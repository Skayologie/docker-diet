📑 Project Specification Document (Cahier des Charges)Project: Docker-Diet (Enterprise Container Optimization & OCI Slimming Engine)1. Project Context & Objectives (Contexte & Objectifs)1.1 ContextIn enterprise environments, container image size is a critical operational liability. Large images introduce deployment bottlenecks across distributed systems, inflate cloud storage ledger items, and broaden the system security attack surface. Existing tools often rely on heavy, external daemons or slow runtime wrappers that are difficult to port across strict, rootless orchestration platforms like Kubernetes or Podman.1.2 Strategic ObjectivesCore Goal: Develop a native, dependency-free CLI optimization engine capable of programmatically stripping dead weight from container layers, reducing image payloads by up to 70% while guaranteeing target execution validity.Architecture-First Design: Prioritize compile-time memory safety, predictable zero-cost abstractions, and direct kernel space-to-user space data structures.Platform Independence: Ensure compatibility with open standards (OCI specs) to allow execution inside standard Docker environments, rootless runtimes, and headless CI/CD runners.2. Functional Scope (Périmètre Fonctionnel)The execution life cycle runs through four sequential architectural phases:[OCI Image / Tarball] ➔ [1. Layer Extraction] ➔ [2. Pure Rust eBPF Profiling] ➔ [3. Deterministic Filtering] ➔ [4. OCI Manifest Synthesis] ➔ [Optimized Image]
2.1 Artifact Ingestion & OCI InspectionAccept a local container image name, a unique hash, or a path to a exported OCI layer tarball: docker-diet analyze <target_image>.Parse the container configuration layer and manifest files natively to map environmental variables (ENV), working directories (WORKDIR), entrypoints, and command history layers without calling an external container engine daemon.2.2 Tracepoint Telemetry & Kernel ProfilingSpawn a monitored container sandbox wrapper.Inject an Aya-based eBPF probe directly into the kernel, pinning hooks to stable tracepoints:tracepoint/syscalls/sys_enter_openat (capturing all filesystem discovery and runtime file loads).tracepoint/syscalls/sys_enter_execve (tracking dynamically spawned child processes or binary executions).Continuous Profiling Loop: Maintain container lifecycle tracking for a defined evaluation period or while an explicit external test matrix (E2E/Integration test script) runs against the isolated application instance.2.3 Strict Deterministic Filtering & Asset PreservationEvaluate the aggregate filesystem changes by cross-referencing the native layer data against the raw bytes logged by the eBPF tracepoints.Safety Guard Rules Engine: Parse a dedicated configuration profile (docker-diet.toml) to explicitly whitelist systemic application paths (e.g., fallback language locales, localization logs, or dynamic modules that operate on conditional exceptions).2.4 Multi-Stage OCI Rebuild & SynthesisProgrammatically compile and output a clean, reproducible multi-stage configuration (Dockerfile.diet).Reconstruct the final layer architecture using a minimal, secure base runtime layer (such as a pure Google Distroless layer or a stripped-down Alpine scratchpad) and copy only verified, trace-validated binary assets and libraries.Output a local optimized image tag, along with a structured metrics readout (Old Size vs. New Size, Asset Drop Rate, Attack Surface Reduction %).3. Technical Specifications (Spécifications Techniques)3.1 Technology Stack SelectionCore System Language: Rust (Latest Stable)Rationale: Compile-time memory safety, deterministic lifecycle (no Garbage Collector interference with profiling metrics), and high-efficiency raw file parsing via std::fs and asynchronous stream manipulation.Kernel Tracing Subsystem: Aya (eBPF framework for Rust)Rationale: Enables a completely unified codebase. Both the kernel bytecode and user-space CLI share exactly identical struct layouts, eliminating type serialization bugs or format conversion lag.Argument & Execution Management: clap (Command Line Argument Parser for Rust) via derive macros for strict, type-safe command orchestration.Container Layer Interfacing: oci-spec (Rust implementation of Open Container Initiative specs), bypassing hard dependencies on standard Docker daemon engine endpoints.3.2 Performance Thresholds (KPIs)Resource Profile: The optimization engine must run with a deterministic memory ceiling (< 50MB RAM during deep file tree analysis).Execution Footprint: The single compiled binary must stand under 15MB to allow fast injection and runtime execution across ephemeral bare-metal or cloud-native CI worker instances.4. Command-Line Interface Layout (CLI UX)The CLI uses strict, explicit subcommands managed through type-driven structures:Bash# Analyze a container target and rebuild the OCI layers
docker-diet analyze --image target-app:latest --config ./docker-diet.toml

# Profile execution duration explicitly on a production payload tarball
docker-diet analyze --tarball ./build/app.tar --duration 120s --output ./optimized.tar

# Verify configuration rules and preview savings metrics without a formal build step
docker-diet dry-run --image target-app:latest
